## Overview

The Entertainment Data Platform provides comprehensive logging and monitoring capabilities across all components. This guide covers how to monitor system health, track data flows, and identify issues.

## Logging Architecture

All services use a centralized logging configuration that outputs structured logs to stdout.

### Log Format

```
%(asctime)s | %(levelname)s | %(name)s | %(message)s
```

Example log output:
```
2024-03-13 10:15:23,456 | INFO | ingestion.producer.kafka_producer | Kafka Producer initialized
2024-03-13 10:15:24,123 | INFO | ingestion.producer.kafka_producer | Message delivered successfully to tmdb.movies.raw, partition 2, offset 1543
```

### Setting Log Levels

Control verbosity using the `LOG_LEVEL` environment variable:

```bash
# Development environment - detailed logs
export LOG_LEVEL=DEBUG

# Production environment - informational logs
export LOG_LEVEL=INFO

# Troubleshooting - all logs
export LOG_LEVEL=DEBUG
```

<Info>
All log output is sent to `stdout` for compatibility with container orchestration platforms like Kubernetes and Docker.
</Info>

## Data Quality Monitoring

### Valid vs Invalid Data Streams

The Stream Processor separates data into valid and invalid streams based on schema validation:

```python src/stream_processor/processor/event_processor.py
def valid_full_schema(df: DataFrame):
    """Check valid schema by adding boolean value to Column "valid_schema"""
    return df.withColumn(
        "valid_schema",
        col("data_type").isNotNull() &
        col("data_label").isNotNull() &
        col("timestamp").isNotNull() &
        col("process_timestamp").isNotNull() &
        col("id_of_data_type").isNotNull()
    )
```

#### Schema Validation Checks

The platform validates the following fields:
- `data_type` - Must be present
- `data_label` - Must be present
- `timestamp` - Must be present
- `process_timestamp` - Must be present
- `id_of_data_type` - Must contain one of: `person_id`, `movie_id`, or `tv_series_id`

<Warning>
Records failing schema validation are written to the **invalid data path** for later analysis. Monitor the invalid paths regularly to identify data quality issues.
</Warning>

### Monitoring Invalid Records

Invalid records are stored in separate Delta Lake tables:

```yaml
# Bronze layer invalid data
s3a://deltalake/bronze/invalid/{data_type}

# Silver layer invalid data
s3a://deltalake/silver/invalid/{data_type}
```

<Steps>
  <Step title="Query Invalid Records">
    Use Spark or Delta Lake to query invalid records and identify patterns.
  </Step>
  
  <Step title="Analyze Failure Reasons">
    Check which validation fields are null or malformed.
  </Step>
  
  <Step title="Fix Upstream Issues">
    Correct data sources or transformation logic to reduce invalid records.
  </Step>
  
  <Step title="Reprocess if Needed">
    Once fixed, reprocess invalid records through the pipeline.
  </Step>
</Steps>

## Storage Monitoring

### Delta Lake Tables

Monitor Delta Lake table growth and health:

```bash
# List all tables in MinIO bucket
aws s3 ls s3://deltalake/bronze/valid/ --endpoint-url=http://localhost:9000

# Check table size
aws s3 ls s3://deltalake/bronze/valid/movies/ --recursive --summarize \
  --endpoint-url=http://localhost:9000

# View Delta Lake transaction log
spark-sql --packages io.delta:delta-core_2.12:3.2.0 \
  -e "DESCRIBE HISTORY delta.\`s3a://deltalake/bronze/valid/movies\`"
```

### ClickHouse Tables

Monitor ClickHouse table statistics:

```sql
-- Check table row counts
SELECT 
    table,
    sum(rows) as total_rows,
    formatReadableSize(sum(bytes)) as total_size
FROM system.parts
WHERE database = 'silver_layer' AND active
GROUP BY table
ORDER BY total_rows DESC;

-- Monitor query performance
SELECT 
    query_duration_ms,
    query,
    memory_usage,
    read_rows
FROM system.query_log
WHERE type = 'QueryFinish'
ORDER BY event_time DESC
LIMIT 10;
```

### Redis Monitoring

Track batch versions and deduplication state:

```bash
# Connect to Redis
redis-cli -h localhost -p 6379

# Check batch versions
GET dedup_batch_version_movie
GET dedup_batch_version_tv_series
GET dedup_batch_version_person

# Monitor Redis memory
INFO memory

# Check key count
DBSIZE
```

<Note>
Redis stores batch version numbers used for incremental processing. If versions are missing, pipelines will abort to prevent data inconsistency.
</Note>

## Error Monitoring

### Common Error Patterns

Monitor logs for these error patterns:

#### Configuration Errors

```
ERROR | Config file not found: /path/to/config.prod.yml
```

**Action**: Verify `APP_ENV` is set correctly and config file exists.

#### Kafka Producer Errors

```
ERROR | Kafka buffer is full, waiting
ERROR | Failed to deliver message ... with error ...
```

**Action**: Check Kafka broker health, increase buffer size, or reduce ingestion rate.

#### Writer Errors

```python
logger.error("Error when write to Clickhouse table: %s.%s", database, table_name)
logger.error("Error when writing Node to Neo4j with label: %s", label)
```

**Action**: Check database connectivity, schema compatibility, and resource limits.

#### Data Validation Errors

```python
logger.error("Invalid data label: %s", data_label)
```

**Action**: Review data source and transformation logic.

## Alerting Recommendations

Set up alerts for critical conditions:

<AccordionGroup>
  <Accordion title="Data Pipeline Alerts">
    - **No data processed in X minutes** - Pipeline stalled
    - **High invalid record ratio (>10%)** - Data quality issue
    - **Checkpoint lag increasing** - Processing falling behind
    - **Kafka consumer lag growing** - Streaming backlog
  </Accordion>

  <Accordion title="Infrastructure Alerts">
    - **Disk usage >80%** - Storage capacity issue
    - **Memory usage >90%** - Resource exhaustion
    - **Service restart detected** - Stability concern
    - **Database connection failures** - Connectivity issue
  </Accordion>

  <Accordion title="Business Logic Alerts">
    - **Missing batch version in Redis** - Pipeline will fail
    - **Schema parsing errors** - Data format changed
    - **Write failures to Gold layer** - Data loss risk
  </Accordion>
</AccordionGroup>

## Log Aggregation

For production environments, aggregate logs using:

- **ELK Stack** (Elasticsearch, Logstash, Kibana)
- **Grafana Loki** with Promtail
- **Datadog** or other APM solutions
- **CloudWatch Logs** (AWS) or **Stackdriver** (GCP)

Example Logstash configuration:

```ruby
input {
  docker {
    type => "entertainment-platform"
  }
}

filter {
  grok {
    match => { "message" => "%{TIMESTAMP_ISO8601:timestamp} \| %{LOGLEVEL:level} \| %{DATA:logger} \| %{GREEDYDATA:message}" }
  }
}

output {
  elasticsearch {
    hosts => ["elasticsearch:9200"]
    index => "entertainment-platform-%{+YYYY.MM.dd}"
  }
}
```
