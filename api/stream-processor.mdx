## Overview

The Stream Processor service consumes events from Kafka topics, validates and enriches them with PySpark, and writes the processed data to Delta Lake tables in MinIO storage. It provides real-time data quality checks and separates valid from invalid records.

## Core Components

### Main Entry Point

<ParamField path="run_stream" type="function">
  Main entry point that sets up Spark streaming from Kafka to Delta Lake.

  **Location**: `src/stream_processor/main.py:22`

  **Process Flow**:
  1. Loads configuration and sets up logging
  2. Creates MinIO bucket if needed
  3. Initializes Spark session with Delta Lake support
  4. Creates streaming queries for each Kafka topic
  5. Processes events and splits into valid/invalid streams
  6. Writes to separate Delta Lake tables
  7. Awaits termination of all queries
</ParamField>

## Usage Examples

### Running the Stream Processor

```python
from stream_processor.main import run_stream

# Start streaming from Kafka to Delta Lake
run_stream()
```

The service will:
- Create Spark session
- Subscribe to Kafka topics
- Process events in micro-batches
- Write to Delta Lake tables
- Run indefinitely until terminated

### Reading from Kafka

```python
from pyspark.sql import SparkSession
from stream_processor.config.settings import load_settings
from stream_processor.sources.kafka_source import read_kafka_stream

# Initialize
settings = load_settings()
spark = SparkSession.builder.appName("KafkaReader").getOrCreate()

# Read stream
df = read_kafka_stream(
    spark=spark,
    settings=settings,
    topic="movie-events"
)

# Show schema
df.printSchema()
```

### Processing Events

```python
from stream_processor.processor.event_processor import process_event

# Read from Kafka
raw_df = read_kafka_stream(spark, settings, "movie-events")

# Process: parse, enrich, validate
processed_df = process_event(raw_df)

# Check schema
processed_df.printSchema()
# Contains: process_timestamp, data_type, data_label, timestamp, 
#          id_of_data_type, valid_schema, raw_df, ...
```

### Writing to Delta Lake

```python
from stream_processor.sinks.delta_lake_sink import delta_lake_sink_dql

# Split and write valid/invalid events
valid_query, invalid_query = delta_lake_sink_dql(
    df=processed_df,
    valid_table="s3a://bronze/movie/valid",
    invalid_table="s3a://bronze/movie/invalid",
    valid_checkpoint="s3a://checkpoints/movie/valid",
    invalid_checkpoint="s3a://checkpoints/movie/invalid"
)

# Queries are now running
print(f"Valid query: {valid_query.status}")
print(f"Invalid query: {invalid_query.status}")

# Wait for termination (blocks)
valid_query.awaitTermination()
```

### Custom Event Processing

```python
from pyspark.sql.functions import col, when, length

# Process events
processed_df = process_event(raw_df)

# Add custom validation
enhanced_df = processed_df.withColumn(
    "has_overview",
    when(length(col("overview")) > 10, True).otherwise(False)
)

# Filter before writing
quality_df = enhanced_df.filter(
    (col("valid_schema") == True) & 
    (col("has_overview") == True)
)
```

## Spark Configuration

The stream processor configures Spark with:

```python
SparkSession.builder \
    .appName("KafkaStreamToDelta") \
    .master("local[*]") \
    .config("spark.sql.extensions", "io.delta.sql.DeltaSparkSessionExtension") \
    .config("spark.sql.catalog.spark_catalog", "org.apache.spark.sql.delta.catalog.DeltaCatalog") \
    .config("spark.databricks.delta.properties.defaults.enableChangeDataFeed", "true") \
    .config("spark.hadoop.fs.s3a.impl", "org.apache.hadoop.fs.s3a.S3AFileSystem") \
    .config("spark.hadoop.fs.s3a.endpoint", settings.sinks.delta_lake.minio_endpoint) \
    .config("spark.hadoop.fs.s3a.access.key", settings.sinks.delta_lake.minio_access_key) \
    .config("spark.hadoop.fs.s3a.secret.key", settings.sinks.delta_lake.minio_secret_key) \
    .config("spark.hadoop.fs.s3a.path.style.access", "true")
```

**Key Features**:
- Delta Lake extensions enabled
- Change Data Feed (CDF) enabled for all tables
- S3A filesystem for MinIO access
- Path-style access for MinIO compatibility

## Data Flow

<Steps>
  <Step title="Consume from Kafka">
    `read_kafka_stream()` creates streaming DataFrame from topic
  </Step>
  
  <Step title="Parse JSON">
    `cast_event()` converts byte array to structured data
  </Step>
  
  <Step title="Enrich">
    `enrich_event()` adds process_timestamp
  </Step>
  
  <Step title="Validate">
    `valid_full_schema()` checks required fields and adds validation flag
  </Step>
  
  <Step title="Split Streams">
    `split_valid_invalid_stream()` separates valid from invalid records
  </Step>
  
  <Step title="Write to Delta Lake">
    `delta_lake_sink()` writes both streams to separate tables
  </Step>
</Steps>

## Event Schema

### Input (from Kafka)

```
value: binary (JSON string)
key: binary
topic: string
partition: int
offset: long
timestamp: timestamp
```

### Output (to Delta Lake - Valid)

```
process_timestamp: timestamp
data_type: string (movie/tv_series/person)
data_label: string (old/new/change)
timestamp: timestamp
id_of_data_type: long
[additional fields from source data]
```

### Output (to Delta Lake - Invalid)

Same as valid, plus:
```
valid_schema: boolean (always false)
raw_df: string (original JSON)
```

## Performance Tuning

<Tip>
**Batch Size**: Controlled by Spark's `spark.sql.streaming.trigger` (default: as soon as possible)

**Parallelism**: Set by `master("local[*]")` - uses all available cores

**Checkpointing**: Essential for fault tolerance; stored in MinIO

**Memory**: Adjust `spark.driver.memory` and `spark.executor.memory` for larger workloads
</Tip>

## Error Handling

<Warning>
**Kafka Connection Failures**: Spark will retry based on Kafka client configuration

**Schema Mismatches**: Records fail validation and go to invalid table

**MinIO Unavailable**: Stream stops; restart from checkpoint when available

**Malformed JSON**: Caught during cast_event; record marked invalid
</Warning>

## Best Practices

1. **Always enable checkpointing** for production workloads
2. **Monitor invalid table** for data quality issues
3. **Use Change Data Feed** for downstream processing
4. **Set appropriate trigger intervals** based on latency requirements
5. **Configure Kafka consumer groups** for parallel processing
